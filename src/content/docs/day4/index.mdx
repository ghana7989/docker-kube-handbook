<Aside type="tip" title="The Big Question">
You have a Kubernetes cluster running. You have Docker images of your application. But Kubernetes does not run Docker containers directly — it runs **Pods**. **What are Pods? How do you deploy your app? And once it is running, how does anyone reach it?**
</Aside>

## What You Will Learn Today

Day 4 answers: **how do we get our application running inside Kubernetes — and how does traffic reach it?**

By the end of today, you will understand:

- **Why** Kubernetes uses Pods instead of running containers directly
- **Why** ReplicaSets exist — maintaining the right number of pods
- **Why** Deployments wrap ReplicaSets — managing updates and rollbacks
- **How** Services route traffic to pods — ClusterIP, NodePort, and service discovery
- **Why** configuration lives outside containers — ConfigMaps and Secrets

## The Story So Far

You have a working Kubernetes cluster (Day 3) and containerized microservices (Day 2). Now it is time to deploy them. But Kubernetes has its own abstractions — Pods, ReplicaSets, Deployments, Services — and each exists for a reason.

Today you will deploy the **Node.js API** into Kubernetes, configure it with environment variables from a ConfigMap, connect it to **Redis** and **PostgreSQL**, and expose it so external traffic can reach it.

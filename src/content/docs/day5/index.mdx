<Aside type="tip" title="The Big Question">
Your application is deployed and serving traffic. Then Monday morning hits — traffic triples. Or worse, you push a bad update and users start seeing errors. **Can your system handle spikes automatically? Can you roll back a bad release without downtime?**
</Aside>

## What You Will Learn Today

Day 5 answers: **how do we make our Kubernetes deployment production-ready?**

By the end of today, you will understand:

- **Why** different deployment strategies exist — rolling, blue-green, canary — and when to use each
- **Why** resource requests and limits matter — and what happens when you skip them
- **How** the Horizontal Pod Autoscaler works — scaling your app based on real metrics
- **Why** a visual dashboard helps — and how to install the Kubernetes Dashboard

## The Story So Far

Over four days, you have containerized a microservice application, built a Kubernetes cluster, and deployed your services into it. Traffic is flowing. But production is not just about "it runs."

Production means:
- **Zero-downtime updates** — deploy new versions without dropping requests
- **Automatic scaling** — handle traffic spikes without manual intervention
- **Resource protection** — prevent one service from starving others
- **Observability** — see what is happening inside your cluster

Today, you make your deployment **production-grade**.

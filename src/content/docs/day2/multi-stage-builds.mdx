## The Problem

You build your Node.js API image. You check the size:

```bash frame="terminal"
docker images my-api
```

It is **450 MB**. But your actual application code is only 2 MB. Where did the other 448 MB come from?

The image contains everything that was needed to **build** the application — compilers, build tools, development dependencies, source maps, test files. None of these are needed at **runtime**. They just bloat the image, slow down deployments, and increase the attack surface.

You need a way to **build** with all the tools but **ship** only what is needed to run.

## The Naive Approach

You could try cleaning up after the build:

```dockerfile title="Doesn't actually help"
FROM node:18
WORKDIR /app
COPY . .
RUN npm install
RUN npm run build
RUN npm prune --production   # Remove devDependencies
RUN rm -rf src test           # Remove source files
CMD ["node", "dist/server.js"]
```

This looks like it should work, but remember — **layers are additive**. The full `node_modules` (with devDependencies) still exists in the `npm install` layer. The source files still exist in the `COPY` layer. You removed them in a later layer, but the image still contains all previous layers. The image is barely smaller.

## Multi-Stage Builds

Multi-stage builds solve this elegantly. You use **two (or more) `FROM` instructions** in a single Dockerfile. Each `FROM` starts a new stage with a clean filesystem. You build in the first stage and copy only the artifacts you need into the final stage.

```dockerfile title="Dockerfile (multi-stage)"
# Stage 1: Build
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Stage 2: Production
FROM node:18-alpine AS production
WORKDIR /app
COPY package*.json ./
RUN npm install --omit=dev
COPY --from=builder /app/dist ./dist
EXPOSE 3000
CMD ["node", "dist/server.js"]
```

### What Happens

1. **Stage 1 (builder)** installs ALL dependencies (including devDependencies like TypeScript, testing tools), compiles the code, and produces the `dist/` folder.

2. **Stage 2 (production)** starts fresh from the same base image, installs only production dependencies, and copies the compiled output from Stage 1.

The final image only contains Stage 2. Stage 1 is **discarded entirely** — its layers do not appear in the final image. The build tools, source code, and devDependencies are gone.

<Aside type="tip" title="The Key Insight">
`COPY --from=builder` is the magic. It reaches into a previous stage's filesystem and copies specific files. Everything else from that stage is thrown away.
</Aside>

## Real-World Size Difference

| Approach | Image Size | Contents |
|----------|-----------|----------|
| Single stage (no cleanup) | ~450 MB | Node.js + all deps + source + build tools + compiled output |
| Single stage (with cleanup) | ~380 MB | Still bloated due to layer history |
| Multi-stage | ~85 MB | Node.js + production deps + compiled output only |

That is an **80% reduction**. In a Kubernetes cluster running 50 replicas, this difference means:
- Faster image pulls on every deployment
- Less storage on every node
- Less data transferred across the network
- Fewer packages for attackers to exploit

## Multi-Stage for Different Languages

The pattern works for any compiled language:

### Go (Even More Dramatic)

```dockerfile title="Go multi-stage"
# Build
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.* ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o server .

# Production — no runtime needed!
FROM alpine:3.19
COPY --from=builder /app/server /server
CMD ["/server"]
```

Go compiles to a static binary — the production stage does not even need Go installed. The final image is **~15 MB** instead of ~300 MB.

### React Frontend

```dockerfile title="React multi-stage"
# Build
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Production — serve static files with nginx
FROM nginx:1.25-alpine
COPY --from=builder /app/build /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

The React build stage produces static HTML/CSS/JS files. The production stage uses nginx to serve them — no Node.js needed at runtime.

## Debugging Multi-Stage Builds

You can build and stop at any stage:

```bash frame="terminal" title="Build only the builder stage"
docker build --target builder -t my-api:debug .
```

This gives you the full build environment. You can run it and inspect what went wrong:

```bash frame="terminal"
docker run -it my-api:debug sh
```

## How This Connects to Our Application

In our microservice stack:
- The **Node.js API** will use a multi-stage build — compile TypeScript in Stage 1, ship only JavaScript in Stage 2
- The **React frontend** will use a multi-stage build — build static files in Stage 1, serve with nginx in Stage 2
- **Redis and PostgreSQL** use official images — no multi-stage needed (they are already optimized)

In [Lab 4](/labs/lab4-containerize-backend/), you will write multi-stage Dockerfiles for both the API and frontend.

## Key Decisions Made

| Decision | Why |
|----------|-----|
| Multi-stage over single stage | 80%+ size reduction. Faster deployments, less storage, smaller attack surface. |
| Separate build and production stages | Build tools should never reach production. They add size and security risk. |
| `--omit=dev` in production stage | Excludes devDependencies (testing tools, linters, compilers) from the final image. |
| nginx for React production | React is static HTML/CSS/JS — no Node.js runtime needed. nginx is 25 MB and purpose-built for serving static files. |

<Aside type="tip" title="The Big Question">
Yesterday you ran **someone else's** containers — nginx, Redis, PostgreSQL. But your application is custom. **How do you turn YOUR code into a container that runs anywhere?**
</Aside>

## What You Will Learn Today

Day 2 answers: **how do we package our own applications into containers?**

By the end of today, you will understand:

- **Why** Dockerfiles exist — and why we don't build images manually
- **How** image layers work — and why they make builds fast
- **Why** multi-stage builds exist — and how they shrink your images from 1GB to 80MB
- **Why** containers lose data when removed — and how volumes solve this
- **How** containers talk to each other — Docker networking explained

## The Story So Far

Yesterday, you learned what containers are and ran pre-built ones. But our microservice application is not on Docker Hub — it is **our code**. Today, we learn to containerize it.

By the end of today, you will have containerized:
- The **Node.js API** backend
- Connected it to **Redis** and **PostgreSQL** containers
- All three running together on a Docker network with persistent data

You should be able to:

- Write a Dockerfile for any Node.js (or similar) application
- Explain **why** each Dockerfile instruction exists
- Optimize builds with layer caching and multi-stage builds

- Run our API + Redis + PostgreSQL stack as connected containers

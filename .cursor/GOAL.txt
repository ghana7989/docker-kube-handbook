This website will be used as the primary teaching material for a real training program.

The website must feel like a professional trainer portal, not product documentation.

The final result should be a fully working Astro + Starlight project with structured documentation pages, labs, and supporting material.

Training Structure

The training is 5 days, but the structure should be story-driven, not command-driven.

Day 1 — Containers and Docker Fundamentals

Goal:
Understand containers and run services with Docker.

Topics to cover:

- Why containers exist
- Containers vs virtual machines
- Docker architecture
- Docker images and containers
- Container lifecycle

Hands-on labs:

Run nginx container

docker run -d -p 8080:80 nginx

Inspect containers

docker ps
docker logs
docker exec -it <container> bash
docker inspect <container>

Students should understand:

- What containers are
- How they run processes
- How networking works

Day 2 — Building Containers

Goal:
Teach how to package an application inside a container.

Topics:

- Dockerfile fundamentals
- Image layers
- Build caching
- Multi-stage builds
- Docker volumes
- Docker networking basics

Hands-on:

Create a Dockerfile for a backend service.

Example:

FROM node:18-alpine
WORKDIR /app
COPY package*.json .
RUN npm install
COPY . .
CMD ["npm", "start"]

Labs:

- Build an image
- Run container from image
- Mount volumes
- Persist database data
- Connect containers using networks

Mini project:

Containerize:

- Backend API
- Redis
- PostgreSQL

Day 3 — Kubernetes Fundamentals

Goal:
Understand how Kubernetes runs containers.

Topics:

- Why Kubernetes exists
- Kubernetes architecture
- Control plane
- Worker nodes
- kube-apiserver
- scheduler
- controller manager
- etcd
- kubelet
- kube-proxy

Students should understand how Kubernetes manages containers across nodes.

Hands-on:

Install Kubernetes cluster using kubeadm.

Cluster example:

control-plane   192.168.56.10
worker-1        192.168.56.11
worker-2        192.168.56.12

Labs:

Initialize cluster

kubeadm init

Join worker nodes

kubeadm join <token>

Install CNI plugin.

Verify cluster:

kubectl get nodes

Day 4 — Running Applications in Kubernetes

Goal:
Deploy and expose applications inside Kubernetes.

Topics:

- Pods
- ReplicaSets
- Deployments
- Services
- ClusterIP
- NodePort
- Service discovery
- ConfigMaps
- Secrets

Hands-on:

Create deployment

kubectl apply -f deployment.yaml

Scale application

kubectl scale deployment api --replicas=3

Expose application

kubectl expose deployment api --type=NodePort

Students deploy the backend service into Kubernetes.

Students learn how services route traffic to pods.

Day 5 — Scaling and Operating Kubernetes Applications

Goal:
Teach production-like Kubernetes capabilities.

Topics:

- Deployment strategies
- Rolling updates
- Blue-green deployment
- Canary releases
- Resource requests and limits
- Metrics server
- Horizontal Pod Autoscaler
- Kubernetes Dashboard

Hands-on:

Update deployment image

kubectl set image deployment api api=app:v2

Create autoscaler

kubectl autoscale deployment api --cpu-percent=70 --min=2 --max=10

Students observe pods scaling automatically.

Operating System Command Tabs

The documentation must support commands for multiple operating systems whenever commands differ.

Supported platforms:
- Linux
- macOS
- Windows

Use tabbed command blocks so users can easily switch between OS-specific instructions.

Examples where this is required:
- Installing Docker
- Installing kubectl
- Installing kubeadm
- Installing container runtime
- Package manager commands
- Path differences
- Shell commands
- Kubernetes cluster setup steps
- Any CLI command that may behave differently across operating systems

Example structure:

Tabs:
Linux
macOS
Windows

Example:

Linux

sudo apt update
sudo apt install docker.io

macOS

brew install docker

Windows

winget install Docker.DockerDesktop

If commands are identical across all systems, do not create separate tabs.

Use a single command block instead.

Important:
- Prefer OS tabs for installation and environment setup steps.
- For Kubernetes cluster labs, assume Linux nodes but still explain how Windows/macOS users can connect using kubectl.
- Ensure commands are accurate for each platform.
- Clearly mention when a step only works on Linux (for example kubeadm cluster setup).

The goal is that learners on Windows, macOS, or Linux can follow the documentation without confusion.
